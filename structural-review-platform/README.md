# 结构项目评审 Web 平台

本平台用于把 `structural-review-agent` 的 M0-M9 审查能力封装为前后端分离的 Web 应用。

## 当前界面规则

## 启动方式

安装依赖后，可分别启动 API 和 Web：

```bash
npm run dev:api
npm run dev:web
```

默认地址：

- API: `http://127.0.0.1:3001`
- Web: `http://127.0.0.1:5173`

验证：

```bash
npm run typecheck
npm run build:web
```

### 项目管理

- 项目支持新建、编辑、删除/移除。
- 删除项目采用软删除，默认项目列表不再展示该项目。
- 软删除不删除审查记录、报告、资料对象记录和审计日志。
- 管理员后续可在回收/已删除视图中查看和恢复项目。

### 固定项目目录

进入项目详情页后，先展示固定目录结构，不直接进入上传页：

```text
MED_000_项目基础信息
MED_001_需求受付
MED_002_ID设计
MED_003_结构方案
MED_004_详细设计
MED_005&007_00_试产
MED_005&007_01_ECN
MED_005&007_02_测试报告
MED_009_量产承认
Trouble list
其他
```

用户点击某个目录后，进入该分类下的操作页。

### 阶段模块清单

每个阶段文件夹下不再只是空容器，而是一个模块清单。模块用于回答“这个阶段的每个关键工程判断条件是否具备”。

模块状态统一为：

```text
未上传 -> 已上传 -> 已解读 -> 需补充/有风险 -> 人工已确认
```

其中 `人工已确认` 必须由用户操作，AI 只能建议，不自动确认。

进入某个阶段文件夹后，页面分三块：

- 模块清单：显示必传/选传、模块状态、资料数量、解读数量、报告数量。
- 资料对象区：对当前模块上传单文件、文件夹、压缩包或批量文件，资料对象允许被多个模块引用。
- 审查操作区：深度解读当前模块、审查整个阶段文件夹、生成正式报告、查看历史报告、查看缺失项、人工确认当前模块。

模块解读卡片包含：

- 模块名称
- 资料状态
- 已识别资料
- 核心内容摘要
- 结构相关判断
- 风险点
- 缺失项
- 需确认问题
- 建议动作
- 置信度
- 是否阻塞当前阶段
- 最低可审查条件是否满足
- 证据链

### 模块准入规则

每个模块都配置专业最低可审查条件，不再只判断“有没有资料”。

例如 `可靠性需求` 至少应包含：

- 测试项目
- 判定标准或验收准则
- 适用场景或使用环境
- 可靠性目标与结构风险的对应关系

如果不满足，平台输出：

```text
资料不足，无法形成可靠性风险判断。
```

### 证据链机制

每条风险、缺失项和判断结论都应能追溯到：

- 项目
- 阶段文件夹
- 模块
- 资料对象
- 资料版本
- 解读记录
- 信息来源类型：明确陈述、上下文推断、视觉判断、未知

### 跨模块冲突检查

M0-M9 综合审查需要检查模块之间的冲突。第一版重点支持：

- `MED_004_详细设计` 下 2D/3D 图纸与 BOM/Part list 的名称对应检查。
- 后续可扩展 ECO 影响范围与 BOM/图纸/测试一致性、测试失败项与 Trouble list 闭环一致性。

模块级审查、阶段级审查和 M0-M9 的分工：

- 模块级审查：读深每个小模块，形成结构化中间结论。
- 阶段级审查：判断该文件夹对应阶段是否完整、是否有风险、是否能推进。
- M0-M9 总审查：优先消费模块级解读结果，再形成正式风险、DFM、ECO、测试闭环、阶段准入和归档判断。

正式报告包含两层结论：

- 模块级摘要：哪些模块已上传、哪些已确认、哪些缺失、哪些阻塞。
- M0-M9 综合结论：正式风险、DFM、ECO、测试闭环、阶段准入、归档建议。

### 分类目录状态

每个目录独立维护状态：

- 未上传
- 已上传
- 待审查
- 审查中
- 已出报告
- 需补充

### 分类目录入口

每个目录页必须提供：

- 上传文件/文件夹
- 查看资料
- 开始审查
- 历史报告
- 缺失项提示

### 资料对象

平台层面把上传内容统一抽象为资料对象，类型包括：

- 单文件
- 文件夹
- 压缩包
- 批量文件

删除动作区分为：

- 从本次审查移除
- 从分类文件夹删除
- 彻底删除

从分类文件夹删除时进入回收站，允许恢复；彻底删除需要管理员权限并记录审计日志。

### 报告命名

报告按以下规则命名：

```text
项目名_审查文件夹名称_审查类型_时间
```

示例：

```text
智能助听器_MED_001_需求受付_完整审查_20260605
```

模块级报告按以下规则命名：

```text
项目名_阶段文件夹名称_模块名称_审查类型_时间
```

示例：

```text
智能助听器_MED_001_需求受付_可靠性需求_模块深度解读_20260608_1036
```

模块报告默认只在平台中作为模块解读卡片展示，用户点击导出或生成正式报告时再汇总进入阶段报告或项目总报告。

## 资料对象版本管理

每个资料对象支持版本历史追踪：

- **当前有效版本**（`effectiveVersion`）：如 `RevA`、`V1.0`，在资料对象基础信息中展示。
- **替换/上传新版本**：进入资料对象详情页 → 版本历史 Tab → 上传新文件，系统自动递增版本标签（RevA → RevB，V1 → V2）。
- **设置有效版本**：在版本历史列表中点击"设为有效版本"，系统更新 `activeVersionId` 和 `effectiveVersion`。
- **历史报告绑定旧版本**：报告生成时记录当时的有效版本，不受后续版本更新影响。
- **新审查默认使用有效版本**：模块深度解读和阶段审查均基于有效版本的资料进行。

## 资料对象详情页（七块布局）

进入资料对象详情页（点击资料名称或"查看详情"按钮），页面分为七个 Tab：

| Tab | 内容 |
|---|---|
| 基础信息 | 名称、类型、MIME、大小、当前有效版本、关联模块、上传时间 |
| 预览 | PDF/图片/文本/Markdown 可直接预览；Excel 展示 sheet 信息；Word/PPT 抽取文本；3D 文件只显示信息 |
| 解析结果 | BOM/Excel 列名识别；Word/PDF 文本抽取说明；3D 文件版本和关联关系 |
| 关联关系 | 资料在 BOM→2D→3D→ECO→测试→Trouble list 链路中的角色；被引用报告列表 |
| 版本历史 | 所有历史版本卡片；支持替换/上传新版本；支持设置有效版本 |
| 审查记录 | 引用了该资料的所有审查报告列表 |
| 操作日志 | 该资料对象的全部操作记录（上传、查看、替换版本、删除、恢复、绑定模块等） |

## BOM / 图纸专项对应检查规则

系统对 `MED_004_详细设计` 等阶段执行 2D/3D 图纸与 BOM/Part list 的名称对应检查：

- **图纸文件名前缀**：`HA1_DE`（对应某零件族）或 `HA1_CB`（另一零件族）
- **BOM Sheet 优先级**：优先匹配与图纸前缀一致的 sheet 名（如 `HA1_DE`）；找不到则自动识别最像 Part list 的 sheet，但结果标记为"需人工确认"
- **列名规则**：BOM 中必须有名为 `2D图纸` 的列和名为 `3D图纸` 的列
- **名称匹配规则**：忽略空格、大小写、扩展名（完全一致才算匹配）

输出六类专项检查结果：

1. BOM 中有记录但未上传图纸
2. 已上传图纸但 BOM 中未登记
3. 名称相似但不完全一致（相似度 60-99%）
4. 版本号冲突（图纸版本与 BOM 中记录版本不一致）
5. 2D/3D/BOM 三者缺一
6. 疑似同一零件但命名不规范，建议修改

操作入口：阶段文件夹页面 → 第三块审查操作区 → "BOM/图纸专项检查"按钮。

## AI 问答模块

### 项目内问答

- **位置**：阶段文件夹页面 → 审查操作区 → "项目AI问答"按钮
- **范围**：仅基于当前项目的资料对象、模块解读卡片、报告和风险清单
- **权限**：仅当前项目有权用户可访问

### 资料库全局问答

- **位置**：页面右下角浮动按钮"资料库问答"（所有页面均可访问）
- **范围**：基于所有已归档项目的历史经验（DFM案例、ECO案例、测试失败闭环等）
- **权限**：只返回用户有权访问的项目资料

### AI 问答边界规则

- AI 只能回答"基于资料可判断 / 部分可判断 / 资料不足"
- **不可以**：自动放行、批准 ECO、关闭测试失败项
- 每个回答必须标注证据来源（项目、阶段、模块、资料、报告、风险编号、解读时间）
- 资料不足时明确告知，不为了回答而回答
- 问答结果标注来源权限范围（仅基于当前项目 / 基于已归档资料库）

## 审计日志扩维（12 类操作）

平台记录以下操作的完整审计日志：

| 操作类型 | 说明 |
|---|---|
| 上传（material.upload） | 资料对象首次上传 |
| 查看（material.view） | 进入资料详情页 |
| 替换版本（material.replace_version） | 上传新版本文件 |
| 修改元数据（material.modify_metadata） | 修改备注、名称等 |
| 删除/移入回收站（material.soft_delete） | 软删除 |
| 恢复（material.restore） | 从回收站恢复 |
| 彻底删除（material.permanent_delete） | 永久删除 |
| 从审查移除（material.remove_from_review） | 仅从本次审查中移除 |
| 绑定模块（material.link_module） | 关联资料到模块 |
| 解绑模块（material.unlink_module） | 取消模块关联 |
| 设置有效版本（material.set_effective_version） | 指定当前有效版本 |
| 人工确认（module.confirm） | 人工确认模块解读 |
| 生成报告（report.generate） | 触发报告生成 |
| AI问答引用（ai_qa.reference） | AI问答引用了项目资料 |

每条审计日志包含：时间戳、projectId、操作类型、目标对象 ID 和描述信息。

## 关联关系视图

资料对象详情页的"关联关系"Tab 和阶段文件夹页面的"关联关系视图"面板，展示资料在完整工程链路中的位置：

```
BOM/Part list → 零件 → 2D图纸 → 3D图纸 → ECO/ECN → 测试报告 → Trouble list
```

系统根据文件名规则自动识别每个资料的角色类型，并以节点-边图形式展示：
- 每个节点显示文件名、引用来源和链接目标
- 节点按类型分组展示（BOM / 2D / 3D / ECO / 测试 / Trouble list / 其他）
- 支持追踪任意零件从 BOM 登记到设计、变更、测试、问题记录的完整链路
