# 结构工程师项目评审 Skill 完整设计说明书

## 1. Skill 定位

本 Skill 面向机械结构工程师在二类医疗器械、电子产品项目中的日常研发工作流。

功能定位：

* 审查项目资料是否齐套、命名和版本是否规范。
* 判断当前项目阶段并提供阶段准入建议。
* 识别结构风险。
* 提取与分析 DFM 问题。
* 评估 ECO/ECN 变更影响。
* 检查测试闭环和会议行动项。
* 输出审查报告、补充问题和下一步建议。

本 Skill 不替代结构工程师做最终设计判断，也不进行真实 CAD/3D 干涉检查或仿真分析。

---

## 2. 使用原则

### 2.1 不让用户给 AI 打工

上传资料后，AI 应自动识别、整理、审查已有资料，用户只补充关键问题。

正确流程：

1. 用户上传项目资料包。
2. AI 自动识别文件夹、文件类型和文件内容。
3. AI 输出已理解信息、缺失信息、不确定信息。
4. 用户只补充 5 到 10 个关键问题。
5. AI 再进行阶段评审、结构风险评审、DFM/ECO/测试闭环审查。

### 2.2 不假装看懂 3D

AI 不应宣称已经完成真实 3D 干涉检查、复杂模具结构判断或强度仿真判断。

AI 必须明确区分：

1. 资料中明确说明的信息。
2. 根据上下文合理推断的信息。
3. 无法判断、需要用户补充或人工确认的信息。

对无法判断的内容，必须标记为“需确认”，不得编造。

### 2.3 先审查资料，再审查设计

每次收到资料包后，Skill 必须先执行文件和资料完整性审查，再进入具体结构风险判断。

推荐顺序：

1. 文件结构审查。
2. 项目阶段识别。
3. 当前阶段资料完整性审查。
4. AI 已理解信息总结。
5. 缺失信息和不确定信息提示。
6. 阶段准入/退出审查。
7. 结构专项风险审查。
8. DFM/ECO/测试闭环审查。
9. 下一步行动建议。

### 2.4 输出报告需可追溯

所有输出报告应标注：

* 项目名称
* 审查模式
* 报告版本
* 生成时间
* 输入文件范围
* AI 判断依据
* 需人工确认内容

## 4. 文件命名规范

### 4.1 通用命名格式

上传文件必须符合公司定义的命名规范。

推荐命名格式：

```text
项目名称_阶段_资料类型_版本号_日期_状态.文件后缀
```

示例：

```text
血压仪项目_结构方案_结构方案说明_RevA_20260603_评审版.pptx
血压仪项目_DFM_供应商DFM报告_RevB_20260605_供应商版.pptx
血压仪项目_ECO_ECO申请单_RevA_20260610_待会签.docx
血压仪项目_DVT_跌落测试报告_RevA_20260615_已完成.pdf
```

```text

```

当文件命名不符合规范时，平台应输出：

1. 原文件名
2. 不符合项
3. 推荐修改名称
4. 是否允许继续分析
5. 是否需要用户确认

示例：

| 原文件名       | 问题                | 推荐文件名                                      | 处理建议    |
| ---------- | ----------------- | ------------------------------------------ | ------- |
| DFM报告.pptx | 缺少项目名、阶段、版本、日期、状态 | 血压仪项目_DFM_供应商DFM报告_RevA_20260603_供应商版.pptx | 建议修改后归档 |

### 4.8 强制规则

对于关键资料，命名不规范时仍可允许 AI 临时分析，但不得进入正式归档状态。

关键资料包括：

1. BOM
2. 图纸包
3. DFM 报告
4. ECO/ECN 文件
5. 测试报告
6. 会签记录
7. 最终归档资料

## 5. 文件版本管理规则

### 5.1 多版本文件处理

当同一阶段、同一资料类型存在多个版本文件时，平台应永远保留最新版本作为当前有效文件。

其余旧版本文件应自动归档到对应目录下的 `old` 文件夹中。

示例：

```text
03_结构方案/
├── 血压仪项目_结构方案_结构方案说明_RevC_20260603_评审版.pptx
└── old/
    ├── 血压仪项目_结构方案_结构方案说明_RevA_20260520_草稿版.pptx
    └── 血压仪项目_结构方案_结构方案说明_RevB_20260528_评审版.pptx
```

### 5.2 最新版本判定规则

优先按照以下顺序判断最新版本：

1. 文件名中的版本号，例如 RevA、RevB、RevC。
2. 文件名中的日期。
3. 文件修改时间。
4. 用户手动指定的有效版本。

当 AI 无法判断哪个文件是最新版本时，应提示用户确认，不得擅自删除、覆盖或忽略文件。

### 5.3 旧版本保留原则

旧版本文件不得删除，只能归档到 `old` 文件夹。

AI 在审查时应默认使用最新版本，但在以下场景中可以参考旧版本：

1. ECO/ECN 变更前后对比。
2. 设计演变追溯。
3. DFM 修改前后确认。
4. 测试失败整改前后对比。
5. 历史项目复盘。

## 6. 支持文件格式与在线编辑规则

### 6.1 支持文件格式

平台应支持结构工程师日常工作常用文件格式，包括但不限于：

1. PPT / PPTX：结构方案、DFM 报告、阶段评审、模具评审。
2. PDF：图纸包、测试报告、供应商报告、会签记录。
3. DOC / DOCX：会议纪要、ECO 说明、设计说明、测试总结。
4. XLS / XLSX：BOM、问题清单、DFM 台账、ECO 影响表、测试数据。
5. PNG / JPG / JPEG：结构截图、爆炸图、截面图、样机照片、问题照片。
6. ZIP / RAR / 7Z：项目资料压缩包。
7. TXT / MD：说明文件、补充说明、README。

### 6.2 在线编辑功能

平台应支持对以下内容进行在线编辑：

1. AI 识别出的项目信息。
2. 文件阶段归类结果。
3. 缺失资料说明。
4. 补充问题回答。
5. 风险等级。
6. DFM 问题状态。
7. ECO 影响分析。
8. 会议行动项。
9. 项目经验总结。

在线编辑后的内容应作为新版本记录，不应直接覆盖历史结果。

## 7. 项目阶段识别规则

Skill 应根据文件夹名称、文件名、文件内容关键词判断当前项目阶段。

### 7.1 需求受付阶段

识别依据：

* PRD
* 用户需求
* 法规需求
* 可靠性需求
* 清洁消毒需求
* 产品定义说明

重点判断：需求是否足以转化为结构设计输入。

### 7.2 ID 设计阶段

识别依据：

* ID 效果图
* 三视图
* 外观尺寸图
* CMF 方案
* ID 评审记录

重点判断：ID 是否具备结构可实现性评审条件。

### 7.3 结构方案阶段

识别依据：

* 结构方案 PPT
* 爆炸图
* 关键截面图
* 初版 BOM
* 结构风险清单

重点判断：方案表达是否完整，是否具备进入详细设计或评审的条件。

### 7.4 详细设计阶段

识别依据：

* 2D 图纸包
* 关键尺寸清单
* 公差分析
* 详细 BOM
* 材料规格书

重点判断：图纸、BOM、材料、版本、关键尺寸是否完整一致。

### 7.5 DFM 阶段

识别依据：

* 供应商 DFM 报告
* DFM 问题清单
* DFM 会议纪要
* 改图记录

重点判断：DFM 问题是否被提取、分类、确认和闭环。

### 7.6 模具试模阶段

识别依据：

* 模具方案评审
* T0/T1/T2 试模报告
* 尺寸报告
* 试模问题清单

重点判断：试模问题是否与 DFM 和结构设计问题闭环。

### 7.7 EVT/DVT/PVT 阶段

识别依据：

* EVT 问题清单
* DVT 测试报告
* PVT 试产报告
* 阶段评审报告

重点判断：测试失败项、试产问题、结构风险是否关闭，是否允许进入下一阶段。

### 7.8 ECO/ECN 阶段

识别依据：

* ECO 申请单
* ECN 通知单
* 变更前后对比
* 变更影响分析
* 验证计划
* 会签记录

重点判断：变更理由、影响范围、验证计划、文件更新和会签是否充分。

### 7.9 项目归档阶段

识别依据：

* 最终 BOM
* 最终图纸包
* DFM 关闭清单
* ECO 关闭清单
* 问题关闭清单
* 设计经验总结

重点判断：项目资料是否完整归档，经验是否沉淀。

## 8. 各阶段最小资料集

当用户资料不完整时，Skill 应根据当前阶段提示缺少的关键资料。

### 8.1 需求受付阶段最小资料

1. PRD 或产品定义说明。
2. 使用场景说明。
3. 可靠性需求。
4. 法规或医疗器械相关要求。
5. 清洁消毒或环境适应性要求。

### 8.2 ID 设计阶段最小资料

1. ID 效果图。
2. 三视图或外观尺寸图。
3. 接口、按键、屏幕、指示灯位置说明。
4. CMF 或材料工艺初步方案。
5. ID 评审结论或待确认问题。

### 8.3 结构方案阶段最小资料

1. 结构方案 PPT。
2. 爆炸图或主要结构组成图。
3. 关键截面图。
4. 初版 BOM。
5. 初步风险清单或评审记录。

### 8.4 详细设计阶段最小资料

1. 2D 图纸包。
2. 详细 BOM。
3. 关键尺寸清单。
4. 材料规格书。
5. 公差或装配间隙分析。

### 8.5 DFM 阶段最小资料

1. 供应商 DFM 报告。
2. 对应结构方案或图纸版本。
3. DFM 问题清单或会议纪要。
4. 改图记录。

### 8.6 ECO 阶段最小资料

1. ECO 申请单或变更说明。
2. 变更前后对比。
3. 受影响零件、BOM 或图纸版本。
4. 变更原因。
5. 验证计划或验证结论。

### 8.7 归档阶段最小资料

1. 最终 BOM。
2. 最终图纸包。
3. DFM 关闭清单。
4. ECO 关闭清单。
5. 测试验证报告。
6. 问题关闭清单。
7. 经验总结。

## 9. 文件审查规则

Skill 应检查以下内容。

### 9.1 文件夹完整性

检查是否存在当前阶段应有的文件夹和文件。

输出：

* 已识别文件夹。
* 缺失文件夹。
* 当前阶段关键缺失资料。

### 9.2 文件命名规范

Skill 应标记：

* 无项目名文件。
* 无阶段字段文件。
* 无资料类型文件。
* 无版本号文件。
* 无日期文件。
* 无状态文件。
* 名称无法识别文件。
* 疑似重复文件。
* 疑似过期文件。

### 9.3 版本一致性

检查：

1. BOM 版本与图纸版本是否一致。
2. DFM 报告引用的图纸版本是否清楚。
3. ECO 变更前后版本是否清楚。
4. 测试报告对应的样机版本是否清楚。
5. 会议纪要中的结论是否对应最新版本。

如果无法判断，标记“版本关系需确认”。

## 10. 结构工程师职责审查维度

Skill 应从结构工程师职责出发审查以下维度。

### 10.1 需求转结构输入

检查：

* 外形尺寸要求。
* 重量要求。
* 使用场景。
* 人机操作方式。
* 患者接触或用户接触部位。
* 清洁消毒要求。
* 防水防尘要求。
* 跌落、振动、运输要求。
* 接口、按键、屏幕、传感器要求。
* 成本、量产规模、装配方式约束。

### 10.2 ID 可实现性

检查：

* ID 三视图是否完整。
* 外观尺寸是否明确。
* 分型线是否可接受。
* 接口、按键、屏幕位置是否可实现。
* CMF 是否有制造和可靠性风险。
* 是否存在清洁死角。
* 是否存在过薄、过尖、难加工结构。

### 10.3 结构方案

检查：

* 外壳拆件方式。
* 固定方式。
* PCBA 固定。
* 电池固定。
* 屏幕固定。
* 按键结构。
* 接口保护。
* 密封结构。
* 装配顺序。
* 可维修性。
* 跌落和运输风险。
* DFM 初步风险。

### 10.4 详细设计

检查：

* 图纸是否齐全。
* 关键尺寸是否定义。
* 公差是否合理。
* 材料是否明确。
* 表面处理是否明确。
* 技术要求是否完整。
* 检验尺寸是否定义。
* 图纸、BOM、3D 版本是否一致。

### 10.5 医疗器械相关结构关注点

检查：

* 清洁消毒兼容性。
* 患者或用户接触部位材料。
* 缝隙、死角、积液风险。
* 外壳破裂后的安全风险。
* 标签、铭牌、追溯标识位置。
* 包装运输保护。
* 是否影响注册资料、风险管理文件、说明书或标签。

## 11. DFM 解析规则

当资料包中包含 DFM PPT、PDF 或问题清单时，Skill 应执行 DFM 解析。

### 11.1 DFM 提取字段

尽量从资料中提取以下字段：

1. DFM 编号。
2. 页码或来源文件。
3. 零件名称。
4. 问题位置。
5. 问题类型。
6. 供应商原始意见。
7. 供应商建议。
8. AI 理解的结构风险。
9. 影响范围。
10. 风险等级。
11. 置信度。
12. 需人工确认问题。
13. 当前状态。

### 11.2 DFM 问题类型

分类包括但不限于：

* 脱模风险
* 缩水风险
* 壁厚风险
* 分型线风险
* 顶针风险
* 滑块/斜顶风险
* 外观风险
* 装配风险
* 强度风险
* 公差风险
* 成本风险
* 周期风险

### 11.3 DFM 输出

DFM 审查应输出：

1. DFM 问题总数。
2. 高/中/低风险数量。
3. 高频问题类型。
4. 需结构工程师优先确认项。
5. 需供应商补充说明项。
6. 可能触发改图或 ECO 的问题。
7. DFM 会议讨论清单。
8. 供应商回复建议。

## 12. ECO/ECN 评审规则

当资料包中包含 ECO/ECN 文件、变更前后对比、BOM 变更或验证计划时，Skill 应执行 ECO 评审。

### 12.1 ECO 必查内容

检查：

1. 变更原因是否清楚。
2. 变更前后是否有对比。
3. 受影响零件是否明确。
4. 图纸版本变化是否明确。
5. BOM 是否变化。
6. 模具是否变化。
7. 生产装配是否变化。
8. 检验标准是否变化。
9. 可靠性验证是否变化。
10. 是否影响法规、注册、风险管理、说明书、标签或包装。
11. 是否有旧料处理方案。
12. 是否有新旧版本切换方案。
13. 是否有会签记录。

### 12.2 ECO 影响分类

按以下类别输出影响分析：

* 结构设计影响
* 模具影响
* BOM/物料影响
* 生产装配影响
* 质量检验影响
* 可靠性验证影响
* 法规注册影响
* 包装标签影响
* 库存和供应链影响
* 售后和已出货产品影响

### 12.3 ECO 输出

输出：

1. 变更摘要。
2. 变更原因。
3. 变更前后对比。
4. 影响范围表。
5. 风险评估。
6. 验证建议。
7. 文件更新清单。
8. 会签部门建议。
9. 旧料处理建议。
10. 是否建议批准。

## 13. 测试和问题闭环审查规则

Skill 应检查 DFM、ECO、测试失败、会议行动项是否形成闭环。

### 13.1 闭环关系

重点检查：

```text
DFM问题 → 是否改图 → 是否试模确认 → 是否关闭
测试失败 → 是否原因分析 → 是否结构变更/ECO → 是否复测通过
ECO变更 → 是否会签 → 是否更新图纸/BOM/检验文件 → 是否完成验证
会议行动项 → 是否有责任人 → 是否有截止日期 → 是否关闭
```

### 13.2 闭环输出

输出：

1. 未关闭 DFM 项。
2. 未关闭 ECO 项。
3. 未关闭测试问题。
4. 未关闭会议行动项。
5. 高风险未闭环事项。
6. 建议优先处理顺序。

## 14. 用户补充问题规则

Skill 不应频繁要求用户补充大量信息。

### 14.1 提问数量

每次最多提出 5 到 10 个关键问题。

### 14.2 提问原则

只问以下三类问题：

1. 会影响阶段判断的问题。
2. 会影响结构风险判断的问题。
3. 会影响 DFM/ECO/测试闭环的问题。

### 14.3 提问格式

每个问题必须包含提问原因。

示例：

```text
1. 当前项目阶段是 EVT、DVT、PVT 还是量产？
   原因：阶段不同会影响 ECO 验证范围和是否需要完整回归测试。

2. 产品是否有酒精擦拭或其他清洁消毒要求？
   原因：这会影响材料选择、缝隙设计、标签耐久性和可靠性验证。

3. DFM-003 是否已接受供应商建议？
   原因：该问题涉及外观面分型线，可能影响 ID 和模具方案。
```

## 15. 置信度和信息来源规则

Skill 输出所有重要判断时，应尽量标注信息来源和置信度。

### 15.1 信息来源分类

1. 资料明确说明。
2. 根据文件名或上下文推断。
3. 图片/截图中疑似可见。
4. 未识别，需用户确认。

### 15.2 置信度等级

* 高：资料中明确写明，或多个文件一致。
* 中：资料中间接体现，但缺少完整说明。
* 低：仅根据图片、文件名或上下文推测。
* 未知：无法判断。

### 15.3 禁止行为

不得：

* 编造资料中没有的尺寸、材料、测试结果或结论。
* 将推测当成事实。
* 宣称已完成 CAD 干涉检查。
* 宣称已完成强度仿真或模流分析。
* 直接替代质量、法规、注册部门做最终结论。

## 17. 平台界面与用户操作说明

### 17.1 工具名称

推荐名称：

```text
结构项目资料审查与风险评审助手
```

备用名称：

```text
结构工程师项目评审助手
```

如果未来包含 DFM、ECO、测试闭环，推荐使用“结构项目资料审查与风险评审助手”，范围更准确。

### 17.2 功能简介

结构工程师项目评审助手用于辅助机械结构工程师完成项目资料审查、阶段评审、结构风险识别、DFM 问题解析、ECO/ECN 变更评审、测试闭环检查和项目归档审查。

用户可上传项目结构资料包、结构方案 PPT、DFM 报告、ECO 资料、测试报告、会议纪要或问题清单。系统将自动识别项目阶段和资料类型，按照结构工程师职责对资料完整性、文件版本、阶段准入条件、结构风险、DFM 问题、ECO 影响范围和测试闭环状态进行审查。

本助手不会替代结构工程师、质量、法规、模具或测试人员做最终判断。对于资料中未明确说明或 AI 无法可靠判断的内容，系统会标记为“需确认”，并只向用户提出影响判断的关键补充问题。

### 17.3 平台卡片简短描述

```text
上传结构项目资料包，自动识别项目阶段，审查资料完整性、结构风险、DFM 问题、ECO 影响和测试闭环状态，并输出补充问题和下一步建议。
```

### 17.4 适用用户

适用用户包括：

1. 结构工程师
2. 项目经理
3. 模具工程师
4. 质量工程师
5. 测试工程师
6. 法规/注册人员
7. 研发主管
8. 供应商质量或供应商管理人员

3. ID 设计结构可实现性评审。

5. ECO/ECN 变更影响分析。

5. 不适用

### 21.2 表格结果导出

平台应支持将表格结果导出为 Excel，便于后续作为项目台账、会议材料或归档资料使用。

---

## 22. 输出格式与导出规则

建议支持以下输出格式：

1. Markdown：适合 ChatGPT 或平台内直接展示。
2. Excel：问题清单、DFM 台账、ECO 影响表。
3. Word：正式审查报告。
4. PPT：评审会议汇报。
5. PDF：归档报告。

第一版建议优先支持：

1. Markdown 报告。
2. Excel 表格。

后续再扩展 Word、PPT、PDF。

---

## 23. 人工确认机制

AI 输出不能直接作为最终结论。

界面上建议设置状态流：

```text
AI 初步识别
结构工程师已确认
质量已确认
法规已确认
项目经理已确认
已关闭
```

尤其是以下内容必须人工确认：

1. ECO/ECN 变更结论。
2. 测试失败关闭结论。
3. 法规或注册影响判断。
4. 高风险结构问题关闭结论。
5. 是否允许进入下一阶段。
6. 是否允许量产或转产。
7. 是否写入最终归档。

---

## 24. 风险等级定义

需要提前定义高、中、低风险，避免 AI 判断标准漂移。

| 风险等级 | 定义                           |
| ---- | ---------------------------- |
| 高    | 可能影响安全、法规、功能、可靠性、量产、注册或项目节点  |
| 中    | 可能影响装配、制造、成本、外观、测试通过率，需要评审确认 |
| 低    | 对设计影响较小，可在常规设计优化或文件更新中处理     |

---

## 25. 阶段准入标准

平台应支持按阶段判断是否允许进入下一阶段。

### 25.1 结构方案进入详细设计

必须满足：

1. 需求输入基本明确。
2. ID 主要尺寸和外观冻结。
3. 主要结构方案明确。
4. 初版 BOM 完成。
5. 关键风险已识别。
6. 无未关闭的高风险需求缺口。

### 25.2 DFM 进入开模

建议满足：

1. 供应商 DFM 报告已完成。
2. 高风险 DFM 问题已评审。
3. 结构修改项已确认。
4. 外观面、分型线、滑块、顶针等关键模具问题已确认。
5. 图纸或 3D 版本明确。
6. 无未关闭的高风险 DFM 问题。

### 25.3 DVT 进入 PVT

建议满足：

1. DVT 测试报告完整。
2. 结构相关失败项已关闭。
3. ECO/ECN 已完成会签和验证。
4. 关键尺寸和装配问题已确认。
5. 试产风险已评估。
6. 无未关闭的高风险可靠性问题。

### 25.4 PVT 进入 MP

建议满足：

1. PVT 试产报告完成。
2. 产线装配问题已关闭。
3. 良率达到目标。
4. 检验标准已确认。
5. 最终 BOM 和图纸版本已冻结。
6. 关键供应商问题已关闭。
7. 无未关闭的高风险量产问题。

---

## 26. 多项目管理规则

### 26.1 多项目资料包上传

平台支持一次上传多个项目资料包。

每个项目必须建立独立的总文件夹，AI 应以项目总文件夹作为项目识别边界，不得将不同项目资料混合分析。

推荐结构：

```text
项目A_结构资料包/
项目B_结构资料包/
项目C_结构资料包/
```

AI 输出报告时，应根据当前操作或识别到的项目文件夹名称区分项目，并分别生成审查结果。

当一次上传多个项目资料包时，AI 应输出：

1. 项目列表。
2. 每个项目的资料完整性。
3. 每个项目的当前阶段判断。
4. 每个项目的风险摘要。
5. 每个项目独立的补充问题。
6. 每个项目独立的审查报告。

### 26.2 历史项目二次审查与复盘

平台允许用户对历史项目进行二次审查或复盘。

历史项目复盘应重点输出：

1. 项目资料是否完整归档。
2. DFM 问题是否闭环。
3. ECO/ECN 是否闭环。
4. 测试失败项是否关闭。
5. 高频结构问题。
6. 可复用经验。
7. 建议沉淀到结构设计规范的内容。
8. 后续类似项目的风险提醒。

---

## 27. AI 解析失败与补充资料规则

### 27.1 解析失败提示

当 AI 无法解析某个文件或某部分内容时，界面应明确提示用户。

提示内容应包括：

1. 无法解析的文件名。
2. 无法解析的原因。
3. 对审查结果的影响。
4. 建议用户补充的资料或操作方式。

示例：

```text
文件：供应商DFM报告_RevA.pptx
问题：第 8 页图片文字无法准确识别。
影响：无法确认供应商对滑块方案的具体建议。
建议：请上传该页高清截图，或手动补充供应商原始意见。
```

### 27.2 用户手动标记

平台应允许用户手动标记以下内容：

1. 文件所属阶段。
2. 文件资料类型。
3. 当前有效版本。
4. 是否为最终版。
5. 是否需要 AI 重点审查。
6. 是否忽略该文件。

### 27.3 补充资料上传

当 AI 提出补充问题后，用户可以通过以下方式补充：

1. 直接在表格中填写答案。
2. 上传补充文件。
3. 上传截图或照片。
4. 标记已有文件作为答案来源。
5. 与 AI 对话补充说明。

AI 应在用户补充资料后重新生成新版本审查报告。

---

## 28. 报告版本、追溯与回溯规则

### 28.1 报告命名规则

所有 AI 输出报告必须带时间戳和版本号。

推荐命名格式：

```text
项目名称_报告类型_版本号_时间戳
```

示例：

```text
血压仪项目_结构项目资料审查报告_V1_20260603_1530
血压仪项目_DFM解析报告_V2_20260604_1015
血压仪项目_ECO影响分析报告_V1_20260605_0900
```

### 28.2 补充资料后的报告版本

每次用户补充资料后，平台应生成新版本报告，而不是覆盖旧报告。

示例：

```text
V1：首次上传资料后生成
V2：用户补充清洁消毒要求后生成
V3：用户补充 DFM 关闭状态后生成
```

### 28.3 历史报告回溯

所有报告均应记录在案，支持回溯历史审查结果。

历史报告至少应保留：

1. 报告版本号。
2. 生成时间。
3. 输入文件范围。
4. AI 识别结果。
5. 用户补充内容。
6. 风险清单。
7. DFM/ECO/测试闭环状态。
8. 当时的结论和建议。

### 28.4 报告差异对比

后续版本可支持报告差异对比。

差异对比重点包括：

1. 新增风险。
2. 关闭风险。
3. 风险等级变化。
4. 新增补充资料。
5. DFM/ECO 状态变化。
6. 阶段准入结论变化。

本部分具体权限策略后续由公司内部确定。

## 30. 知识沉淀与复用规则

### 30.1 高频问题点沉淀

Skill 输出的高频问题、DFM 问题类型、ECO 原因分析、测试失败原因和项目经验，应沉淀到知识库的单独栏目中。

建议栏目名称：

```text
高频问题点
```

也可细分为：

1. 结构高频问题点。
2. DFM 高频问题点。
3. ECO 高频原因。
4. 测试失败高频问题。
5. 供应商高频问题。
6. 阶段评审常见缺失项。
7. 设计规范更新建议。

### 30.2 高频问题点字段

每条高频问题建议包含以下字段：

| 字段       | 说明                                          |
| -------- | ------------------------------------------- |
| 问题编号     | 系统自动生成                                      |
| 项目名称     | 来源项目                                        |
| 阶段       | 需求 / ID / 结构方案 / DFM / ECO / DVT / PVT / 归档 |
| 零件名称     | 上壳、下壳、按键、支架等                                |
| 问题类型     | 结构、DFM、ECO、测试、装配、供应商等                       |
| 问题描述     | 问题现象或风险描述                                   |
| 原因分析     | 初步或最终原因                                     |
| 解决措施     | 已采取措施                                       |
| 是否触发 ECO | 是 / 否                                       |
| 是否复测通过   | 是 / 否 / 不适用                                 |
| 是否写入设计规范 | 是 / 否 / 建议写入                                |
| 来源文件     | DFM 报告、测试报告、会议纪要等                           |
| 置信度      | 高 / 中 / 低                                   |
| 状态       | 待确认 / 已确认 / 已沉淀                             |

### 30.3 项目经验总结

平台应支持项目经验总结自动整理。

项目结束或归档审查时，AI 应自动输出：

1. 本项目主要结构风险。
2. 本项目主要 DFM 问题。
3. 本项目主要 ECO 原因。
4. 本项目测试失败及整改经验。
5. 本项目供应商沟通经验。
6. 对后续项目的预防建议。
7. 建议更新到结构设计规范的条目。
8. 可复用设计经验。
9. 不建议复用的设计方案或风险点。

## 31. 异常处理、用户反馈与规则迭代

### 31.1 AI 输出错误反馈

当 AI 输出明显错误、遗漏或误判时，平台应提供用户反馈入口。

用户可标记：

1. 识别错误。
2. 文件阶段分类错误。
3. 文件版本判断错误。
4. 风险等级错误。
5. DFM 问题提取错误。
6. ECO 影响分析错误。
7. 测试闭环判断错误。
8. 输出内容不完整。
9. 需要新增规则。

### 31.2 用户与 AI 对话功能

平台应开通用户与 AI 对话功能。

用户可以通过对话完成：

1. 纠正 AI 识别结果。
2. 补充项目背景。
3. 解释特殊文件。
4. 追问某个风险原因。
5. 要求 AI 重新分析某一阶段。
6. 要求 AI 生成 DFM 台账。
7. 要求 AI 生成 ECO 影响分析。
8. 要求 AI 生成会议纪要。
9. 要求 AI 更新审查规则建议。

### 31.3 规则学习与优化机制

Skill 应具备规则优化机制。

当用户多次纠正同类问题时，平台应支持将该反馈沉淀为规则更新建议。

规则更新建议应包括：

1. 原规则。
2. 用户反馈。
3. 建议修改内容。
4. 适用阶段。
5. 适用文件类型。
6. 是否影响输出模板。
7. 是否需要管理员确认。

### 31.4 规则更新审批

AI 不应自行修改正式规则。

正式规则更新应由指定人员确认后生效。

建议状态：

1. AI 建议更新。
2. 结构工程师确认。
3. 管理员批准。
4. 规则已发布。
5. 规则已归档。

## 32. 第一版 Skill 建议范围

第一版不建议做得过大，应优先覆盖以下能力：

1. 项目文件夹结构审查。
2. 当前阶段识别。
3. 资料完整性检查。
4. 结构方案 PPT 预审。
5. DFM PPT 问题提取。
6. ECO 影响分析。
7. 测试/会议/问题闭环检查。
8. 固定格式审查报告输出。
9. 表格化结果输出。
10. 用户补充问题与新版本报告生成。

暂不覆盖：

1. 自动读取 CAD 原生 3D 文件。
2. 自动完成真实干涉检查。
3. 自动完成强度仿真。
4. 自动完成模流分析。
5. 自动连接 PLM、ERP、QMS 系统。

这些可以作为后续 Agent 或企业系统集成阶段再做。
